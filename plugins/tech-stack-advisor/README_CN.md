# Tech Stack Advisor - 技术栈顾问插件

智能技术选型顾问插件,提供基于最新资料的技术栈选型建议、深度对比分析和实施指导。

## 功能特性

### 双模式支持

#### 1. 快速咨询模式 (Skill)
适合快速获取技术建议和主流推荐:
- 基于最佳实践的快速推荐
- 2-3 个主流方案的简洁对比
- 优缺点和适用场景分析
- 无需网络调研,响应迅速

**使用场景**:
- 已有明确候选方案,需要快速对比
- 寻求主流技术推荐
- 基于常见场景的选型建议

**调用方式**:
```
使用 Skill 工具调用 tech-stack-advisor:tech-stack-advisor
```

#### 2. 深度选型模式 (Agent)
适合需要详细调研和数据支持的选型决策:
- 在线调研最新技术动态
- 量化数据指标收集(GitHub Stars、NPM 下载量等)
- 多维度深度对比分析
- 详细的实施建议和迁移成本评估
- 完整的选型报告生成

**使用场景**:
- 需要查询最新技术趋势
- 要求数据驱动的决策支持
- 需要详细的多方案对比报告
- 评估迁移成本和风险

**调用方式**:
```
使用 Task 工具调用,指定 subagent_type="tech-stack-advisor:tech-stack-advisor"
```

## 选型报告内容

深度选型模式会生成包含以下内容的完整报告:

### 1. 基础对比分析
- 功能完整性对比
- 性能指标对比
- 开发体验对比
- 生态系统对比
- 成熟度和稳定性分析

### 2. 量化数据指标
- GitHub Stars、Forks、Contributors
- NPM 周下载量、总下载量
- 最后更新时间、Release 频率
- Open Issues 数量、Issue 关闭率
- 社区活跃度数据

### 3. 实施建议
- 技术准备(环境配置、依赖管理)
- 团队准备(学习路径、培训计划)
- 风险缓解措施
- 监控指标设定

### 4. 迁移成本评估
- 详细的工作量评估
- 迁移策略建议(渐进式 vs 一次性)
- 长期成本分析
- ROI 评估

## 工作流程

### 快速咨询流程
1. 收集基本信息(项目类型、技术范围、约束条件)
2. 提供 2-3 个主流方案对比
3. 给出明确推荐和理由

### 深度选型流程
1. **需求收集**: 全面了解项目背景、团队情况、约束条件
2. **在线调研**: 查询官方文档、社区讨论、性能测试、应用案例
3. **数据收集**: 获取量化指标和最新动态
4. **对比分析**: 多维度系统化对比
5. **生成报告**: 输出完整的选型报告

## 使用示例

### 示例 1: 快速咨询
```
用户: 我要开发一个企业级后台管理系统,前端框架应该选 React 还是 Vue?

系统: 使用 Skill 调用 tech-stack-advisor

输出: 简洁的对比分析,包括两者的核心优势、适用场景和推荐结论
```

### 示例 2: 深度选型
```
用户: 我需要为一个大型电商项目选择后端框架,需要详细的对比报告

系统: 使用 Task 调用,subagent_type="tech-stack-advisor:tech-stack-advisor"

输出:
1. 收集项目需求(规模、性能要求、团队情况等)
2. 在线调研候选方案(Express、Fastify、Nest.js 等)
3. 收集性能数据、社区数据、应用案例
4. 生成完整的选型报告,包括多维度对比、实施建议、迁移评估
```

## 注意事项

1. **快速模式的局限性**
   - 基于截至 2025-01 的知识
   - 不包含最新的网络调研
   - 适合常见场景,不适合特殊需求

2. **深度模式的时效性**
   - 数据采集时间会明确标注
   - 建议 3 个月内参考
   - 技术快速变化时需要重新调研

3. **推荐的前提条件**
   - 所有推荐都基于特定的前提条件
   - 需要结合实际情况判断
   - 不存在"银弹"方案

## 最佳实践建议

### 何时使用快速模式
- 时间紧迫,需要快速决策
- 技术选型范围明确,候选方案清晰
- 项目规模较小,风险可控
- 团队对候选技术已有一定了解

### 何时使用深度模式
- 关键技术决策,影响长远
- 项目规模大,迁移成本高
- 需要说服团队或管理层
- 候选方案多,差异不明显
- 技术较新,缺乏实践经验

### 如何提高选型质量
1. **提供详细的需求信息**
   - 项目背景和目标
   - 技术约束和限制
   - 团队情况和能力

2. **明确关键决策因素**
   - 哪些因素最重要(性能?学习曲线?生态?)
   - 可接受的权衡是什么
   - 不可妥协的底线在哪里

3. **考虑长期因素**
   - 不仅看当前需求,也要考虑未来扩展
   - 评估团队的长期学习和维护能力
   - 关注技术的发展趋势
